# Fitnes App — Roadmap & Checklist

Файл для отслеживания всех запланированных доработок приложения.
Обновляется по мере выполнения. Статус: `[ ]` — не начато, `[~]` — в процессе, `[x]` — готово.

## Блок 0 — Подписки (фундамент)
> **Почему первым:** почти все новые функции зависят от тарифа пользователя. Без подписочного слоя придётся захардкодить «если Про» в каждом месте и потом переписывать всё заново.

- [x] **Определить тарифы: Free / Стандарт / Про**
  Тарифы и ограничения зафиксированы в спеке и приложении: `free`, `standard`, `pro`. Это стало основой для paywall-логики и feature-gating.

- [x] **Таблица `subscriptions` в Supabase**
  Таблица создана и применена в Supabase: хранит `user_id`, `plan` (`free/standard/pro`), `expires_at`, `status` (`active/cancelled`), `created_at`, `updated_at`. RLS настроен, записи управляются вручную через Dashboard в рамках MVP.

- [x] **Хук `useSubscription()` — проверка тарифа в любом компоненте**
  Хук реализован: возвращает текущий план, флаги `isFree / isStandard / isPro / isPaid` и helper `canAccess(...)`. Отсутствие записи в `subscriptions` трактуется как `free`.

- [x] **Экран выбора / апгрейда тарифа (paywall UI)**
  Экран сравнения тарифов реализован. Он открывается из профиля и из закрытых функций. Текущий формат — MVP без встроенной оплаты: вместо billing-интеграции используется ручная активация через поддержку.

- [x] **Заглушки для Pro-функций**
  Стандартизированный `PaywallBanner` добавлен и уже используется для закрытых сценариев Блока 0. Вместо ошибки пользователь получает понятное приглашение перейти на нужный тариф.

## Блок 1 — Главная (редизайн + новые секции)

### 1.1 Система баллов
> Геймификация: пользователь зарабатывает баллы за активность и может обменять их на скидку подписки. Мотивирует регулярно тренироваться.

- [x] **Таблица `points_log` в Supabase**
  Каждое начисление — отдельная запись: `user_id`, `amount` (сколько баллов), `reason` (например «завершил тренировку», «7 дней подряд»), `created_at`. Баланс считается суммой всех записей для пользователя.

- [x] **Автоматическое начисление баллов за тренировку**
  При завершении тренировки (нажатие «Завершить» в `[id].tsx`) — триггер в Supabase или Edge Function записывает начисление в `points_log`. Количество баллов зависит от сложности тренировки: начинающий = 10, средний = 20, продвинутый = 30.

- [x] **Виджет баланса баллов на главной**
  Компактная карточка на главном экране: иконка звезды/монеты, текущий баланс баллов, прогресс-бар до следующего порога скидки. По тапу открывается экран «Мои баллы».

- [x] **Экран «Мои баллы» — история начислений**
  Список всех начислений с датой, причиной и количеством баллов. Наверху — текущий баланс и доступные скидки. Дизайн в стиле приложения (тёмные карточки, фиолетово-синяя палитра).

- [ ] **Механика обмена баллов на скидку**
  Пользователь может применить баллы при оплате подписки: например 500 баллов = -10% на следующий месяц. На экране баллов кнопка «Применить к подписке» — создаёт промо-код или передаёт флаг в платёжную систему. Отложено до появления реальной оплаты.

### 1.2 Прогресс
> Визуальные графики дают пользователю понимание своей динамики — один из главных мотиваторов продолжать тренировки.

- [x] **График тренировок по дням/неделям**
  Столбчатый или линейный график (react-native-chart-kit) — сколько тренировок за последние 7/30 дней. Данные берутся из таблицы `workout_logs`. Позволяет увидеть регулярность и провалы.

- [x] **График динамики веса (с экраном записи нового веса)**
  Линейный график изменения веса со временем. Пользователь вводит вес вручную (или из HealthKit на iOS). Хранится в таблице `weight_log` (user_id, value, measured_at). Видно за месяц / 3 месяца / всё время.

- [~] **График КБЖУ по дням**
  Горизонтальные полосы калорий, белков, жиров, углеводов за последние 7 дней. Данные из `food_log` (блок 2). Позволяет увидеть насколько стабильно соблюдается рацион. Заглушка реализована (NutritionChartPlaceholder). Активируется в Блоке 2 вместе с трекером еды.

### 1.3 Нормы воды и КБЖУ
> Пользователь сразу видит на главной сколько ещё нужно выпить и съесть сегодня — без перехода на другие экраны.

- [x] **Расчёт дневной нормы воды**
  Норма берётся из `user_settings.water_intake_goal_ml`. Если не задана — дефолт 2000 мл. Никакого пересчёта на клиенте нет.

- [x] **Расчёт КБЖУ по параметрам пользователя**
  Нормы берутся из `user_settings` (`daily_calories`, `daily_protein_g`, `daily_fat_g`, `daily_carbs_g`). Источник истины — только `user_settings`.

- [x] **Кольцевые прогресс-индикаторы на главной**
  Реализован `HomeProgressSection` с `ProgressRing`: большое кольцо воды + четыре кольца КБЖУ. Фактические данные — из `water_log` и `nutrition_log`.

- [x] **Обновление в реальном времени**
  Supabase Realtime подключён для `water_log` и `nutrition_log`. Инвалидация через TanStack Query без перезагрузки экрана.

## Блок 2 — Таб «Трекер» (замена вкладки Scanner)
> Текущая вкладка Scanner делает только одно — открывает камеру. Теперь это будет полноценный дневник питания и активности, а камера станет одним из инструментов внутри.

- [ ] **Активировать экраны добавления воды/еды и убрать индикаторы "В разработке"**
  Завершить временные маршруты из Блока 1.3: подключить реальные формы, заменить placeholder-экраны и убрать UI-маркеры, которые сейчас честно сообщают пользователю, что функция ещё в работе.

### 2.1 Навигация
- [ ] **Переименовать вкладку Scanner → «Трекер»**
  В файле `app/(tabs)/_layout.tsx` изменить название и иконку вкладки. Файл `scanner.tsx` переименовать в `tracker.tsx`. Обновить все ссылки.

- [ ] **Обновить иконку вкладки**
  Вместо иконки камеры — иконка дневника или графика (например `BookOpen` или `BarChart2` из lucide-react-native), отражающая суть трекера.

### 2.2 Трекер воды
- [ ] **Быстрое добавление воды**
  На экране трекера — блок воды с кнопками быстрого добавления: 100 мл, 200 мл, 300 мл, и поле для произвольного значения. Одно нажатие = запись в `water_log` (user_id, amount_ml, logged_at).

- [ ] **История воды за день**
  Список всех добавлений воды за сегодня с временными метками. Можно удалить ошибочную запись. Итог за день показывается жирным вверху блока.

- [ ] **Кольцевой индикатор прогресса воды**
  Большое кольцо или дуга: выпито X мл из Y нормы. Заполняется пропорционально. При достижении нормы — зелёный цвет и анимированное поздравление.

### 2.3 Фуд-трекер
- [ ] **Таблица `food_log` в Supabase**
  Каждый приём пищи — отдельная запись: `user_id`, `date`, `meal_type` (завтрак/обед/ужин/перекус), `name` (название блюда), `kcal`, `protein`, `fat`, `carbs`, `logged_at`. Индексы по user_id + date для быстрой выборки.

- [ ] **Добавление еды вручную**
  Поиск блюда по названию (база блюд в Supabase или внешнее API). Если не найдено — ввод КБЖУ вручную. Выбор типа приёма пищи (завтрак / обед / ужин / перекус).

- [ ] **Просмотр КБЖУ по дням**
  Экран истории: календарь или список дней, суммарные КБЖУ за каждый день. Можно переключаться между днями. При выборе дня — список всех блюд с разбивкой по приёмам пищи.

- [ ] **AI-сканер еды через камеру**
  Иконка фотоаппарата в правом верхнем углу трекера. По нажатию — открывается камера, делается фото. Фото отправляется в уже готовую Edge Function `analyze-food-vision`, которая возвращает название блюда и КБЖУ. Пользователю показывается карточка с результатом и кнопкой «Добавить в дневник».

- [ ] **Стандарт: общий набор блюд и ручной ввод**
  Для пользователей на базовом тарифе — поиск по общей базе блюд или ручной ввод. Без персонализации. Полный доступ к трекеру воды и КБЖУ.

- [ ] **Про: персонализированное меню**
  Для Pro-пользователей — дополнительный раздел «Рекомендованное меню» с планом питания, сгенерированным AI под параметры пользователя (цель, КБЖУ, предпочтения из онбординга). Меню обновляется раз в неделю через Edge Function.

### 2.4 Сводка дня
- [ ] **Экран «Сегодня» — сводка всех показателей**
  Верхняя секция трекера: дата, итоговые КБЖУ за сегодня (кольца или полосы), вода (прогресс к норме), была ли тренировка. Быстрый обзор одним взглядом. Ниже — детальные секции воды и питания.

## Блок 3 — Тренировки (рабочий календарь)
> Сейчас календарь отображается, но не показывает реальные данные. Пользователь не видит историю и не может планировать — ключевая функция трекера тренировок.

- [ ] **Диагностика: почему календарь не работает**
  Разобраться в текущей реализации `workouts.tsx` — что рендерится, какие данные запрашиваются, почему ячейки пустые. Проверить есть ли данные в `workout_logs` и правильно ли они читаются.

- [ ] **Отображение завершённых тренировок на календаре**
  Каждый день, когда пользователь завершил тренировку — ячейка окрашивается (зелёная точка или заливка). Данные из таблицы `workout_logs` (user_id, workout_id, completed_at). Выборка за текущий месяц.

- [ ] **Клик по дню — детали тренировки**
  При нажатии на день с тренировкой — появляется нижний лист (bottom sheet) или переход на экран с деталями: название тренировки, длительность, упражнения, заработанные баллы.

- [ ] **Визуальные маркеры: выполнено / пропущено / сегодня**
  Цветовая система: зелёный = выполнено, серый = пропущено (прошедший день без тренировки), синий контур = сегодня, пустой = будущий день. Легенда под календарём.

- [ ] **Отображение запланированных тренировок**
  Если у пользователя есть план (Pro-функция или ручное планирование) — будущие дни с запланированной тренировкой отображаются с особым маркером. Опционально для первого релиза.

## Блок 4 — Профиль (новые секции)

### 4.1 Справочник
> Контентная секция: статьи, советы, новости, челленджи. Делает приложение полезным даже в дни без тренировок и повышает вовлечённость.

- [ ] **Таблица `knowledge_base` в Supabase**
  Хранит контент для справочника: `id`, `category` (food / training / nutrition / news / challenge), `title`, `body` (текст статьи, Markdown), `image_url`, `published_at`, `is_published`. Контент добавляет администратор.

- [ ] **Экран «Справочник» с категориями**
  Горизонтальный скролл категорий (Еда / Тренировки / Питание / Новости / Челленджи) и сетка карточек ниже. При выборе категории — фильтрация. Дизайн: крупные карточки с фото, заголовок, кол-во минут на чтение.

- [ ] **Карточки статей**
  Каждая карточка: обложка (фото), категория-бейдж, заголовок, краткое описание (2 строки). Стиль — тёмные карточки с мягкими тенями, соответствует общей палитре приложения.

- [ ] **Экран детали статьи**
  Полный текст статьи с поддержкой Markdown (заголовки, жирный, списки). Верхняя обложка, заголовок, категория, дата. Кнопка «Назад» и кнопка «Поделиться».

### 4.2 Фото прогресса (до/после)
> Пользователь фиксирует визуальный прогресс фотографиями. Еженедельные уведомления мотивируют не забывать. Со временем — мотивирующая галерея собственного пути.

- [ ] **Таблица `progress_photos` в Supabase**
  Хранит фото: `user_id`, `photo_url` (ссылка в Storage), `taken_at`, `note` (необязательная подпись). RLS: пользователь видит только свои фото.

- [ ] **Приватный Storage bucket для фото**
  Bucket `progress-photos` в Supabase Storage с RLS-политикой: read/write только для владельца. Фото не доступны анонимно. При загрузке — изображение сжимается на клиенте перед отправкой.

- [ ] **Экран «Мой прогресс» — галерея фото**
  Сетка 2×N фотографий, отсортированных по дате (новые сверху). При нажатии — полноэкранный просмотр с датой и заметкой. Первое фото помечается «Старт», следующие — датой.

- [ ] **Загрузка нового фото**
  Кнопка «+» или FAB на экране прогресса. Предлагает выбор: сделать фото прямо сейчас или выбрать из галереи. После выбора — поле для заметки (необязательно) и кнопка «Сохранить».

- [ ] **Еженедельное push-уведомление**
  Каждое воскресенье (или другой день по настройке) в выбранное время — уведомление «Запечатлей результат! Зафиксируй свой прогресс за эту неделю 📸». Реализуется через expo-notifications с локальным расписанием.

### 4.3 Настройки уведомлений
> Пользователь сам выбирает какие напоминания получать и в какое время — это уважение к его расписанию и снижение раздражения от нежелательных пушей.

- [ ] **Экран настроек уведомлений**
  Список типов уведомлений: Вода (напоминание пить), Тренировки (время заниматься), Прогресс-фото (еженедельный), Общие (новости, челленджи). Для каждого — тогглер вкл/выкл и выбор времени (TimePicker).

- [ ] **Включение/выключение каждого типа + время**
  При включении типа — запрашивается разрешение (если ещё не дано). Локальные уведомления планируются через expo-notifications. При выключении — уведомления этого типа отменяются.

- [ ] **Сохранение настроек в `user_settings`**
  Настройки уведомлений хранятся в Supabase в таблице `user_settings` (колонки `notifications_water`, `notifications_workout`, `notifications_photo`, и т.д. — boolean + time). Синхронизируются при входе на новое устройство.

### 4.4 Имя пользователя
> Персонализация: вместо email или «Привет!» пользователь видит своё имя. Простая деталь, сильно влияющая на ощущение «своего» приложения.

- [ ] **Поле `username` в таблице `profiles`**
  Добавить колонку `username` (text, nullable) в таблицу `profiles`. Миграция Supabase. Уникальность не обязательна — это отображаемое имя, не логин.

- [ ] **Редактирование имени в настройках профиля**
  В блоке настроек профиля — поле «Имя», которое можно отредактировать. При сохранении — обновляется запись в `profiles`. Валидация: не пустое, не длиннее 50 символов.

- [ ] **Отображение имени на главной и в профиле**
  На главном экране: «Привет, Даниил! 👋» вместо безликого приветствия. В шапке профиля — имя под аватаром. Если имя не задано — показывается «Атлет» по умолчанию.

## Блок 5 — Чат (разветвление по тарифу)
> На базовом тарифе — ИИ-тренер как сейчас. На Про — чат с реальным тренером, который видит историю пользователя и отвечает лично. Это главное конкурентное преимущество тарифа Про.

- [ ] **Стандарт: ИИ-чат без изменений**
  Текущая реализация ИИ-чата остаётся как есть для пользователей без Pro-подписки. Возможно — лимит на количество сообщений в день для Free-тарифа.

- [ ] **Про: чат с живым тренером**
  Отдельный канал Supabase Realtime для Pro-пользователей. Тренер подключается через будущую web-панель (блок 6) и видит историю переписки. Сообщения хранятся в `trainer_messages` (user_id, trainer_id, content, is_from_user, created_at).

- [ ] **Маршрутизация по тарифу**
  При открытии вкладки «Тренер» — проверка тарифа через `useSubscription()`. Стандарт → ИИ-чат. Про → чат с тренером. Переключение без перезапуска приложения при смене тарифа.

- [ ] **UI: индикация «Вы общаетесь с тренером»**
  Шапка чата меняется: вместо «🤖 ИИ-тренер» — аватар и имя реального тренера. Бейдж «PRO» в углу. Визуально понятно, что это живой человек.

- [ ] **Статус тренера (онлайн / не в сети)**
  Зелёная точка — тренер онлайн и отвечает в течение часа. Серая — не в сети, ответит позже. Статус устанавливается тренером из web-панели.

- [ ] **Push-уведомление при ответе тренера**
  Когда тренер отправляет сообщение — пользователь получает push-уведомление «Тренер ответил на ваш вопрос». Реализуется через Supabase Edge Function + Expo Push Notifications API.

## Блок 6 — Web-панель тренера (будущее, отдельный проект)
> Отдельный веб-сайт (Next.js) на том же Supabase backend. Тренер управляет клиентами, отвечает в чатах, видит прогресс. Разрабатывается после того как мобильная часть готова.

- [ ] **Список клиентов с параметрами**
  Таблица всех Pro-пользователей: имя, фото, основные параметры (цель, вес, рост), дата последней тренировки, активность. Поиск и сортировка.

- [ ] **Просмотр прогресс-фото клиента**
  Тренер видит галерею фото прогресса клиента (до/после). Помогает оценить результат и дать персональный совет.

- [ ] **Чат с Pro-клиентами**
  Интерфейс чата для тренера: список диалогов, непрочитанные сообщения, ответы. Те же данные что и в мобильном приложении через Supabase Realtime.

- [ ] **Назначение тренировок и планов питания**
  Тренер может назначить клиенту конкретный план тренировок или план питания из существующих в базе. Клиент видит это в мобильном приложении.
