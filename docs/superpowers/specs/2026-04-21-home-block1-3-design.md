# Блок 1.3 — Главная: Нормы воды и КБЖУ

**Дата:** 2026-04-21  
**Статус:** Утверждён  
**Этап:** Полноценный интерактивный блок на главной с placeholder-экранами для будущего Блока 2

---

## Контекст

В `Redesign.md` следующий этап после Блока 0 — `Блок 1`, а первым подблоком для реализации выбрана `1.3 Нормы воды и КБЖУ`. Цель этого этапа — заменить текущие секции воды и КБЖУ на главной на единый интерактивный блок прогресса с реальными данными из Supabase, кольцами прогресса и realtime-обновлением.

При этом пользовательские экраны добавления воды и еды пока не реализуются полностью. Вместо них на этом этапе добавляются отдельные маршруты-заглушки с явным статусом `В разработке`, чтобы UX был честным, а архитектура уже была готова к Блоку 2.

---

## Секция 1 — Архитектура

### Подход

Для реализации используется отдельный feature-срез, а не точечный патч `HomeScreen`.

Состав среза:
- миграция `water_log` в Supabase
- data-layer для воды и realtime-обновления питания
- UI-компоненты блока дневного прогресса и круговых индикаторов
- отдельные placeholder-экраны добавления воды и еды
- упрощённый `HomeScreen`, который только собирает новый блок из готовых сущностей

### Границы ответственности

- `water_log` хранит только фактические записи воды
- `nutrition_log` остаётся источником фактических данных по КБЖУ
- `user_settings` остаётся единственным источником норм
- главная не пересчитывает нормы локально и не хранит воду в `AsyncStorage`
- все ещё неготовые действия ведут на отдельные маршруты с явной маркировкой `В разработке`

## Секция 2 — Данные

### Таблица `water_log`

```sql
id         uuid        PRIMARY KEY DEFAULT gen_random_uuid()
user_id    uuid        NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE
amount_ml  integer     NOT NULL CHECK (amount_ml > 0)
logged_at  timestamptz NOT NULL DEFAULT now()
created_at timestamptz NOT NULL DEFAULT now()
```

### RLS политики

- `SELECT`: пользователь читает только свои записи (`auth.uid() = user_id`)
- `INSERT`: пользователь создаёт только свои записи (`auth.uid() = user_id`)
- `UPDATE/DELETE`: на этом этапе не открываются

### Источник норм

Нормы берутся только из `user_settings`:
- `water_intake_goal_ml`
- `daily_calories`
- `daily_protein_g`
- `daily_fat_g`
- `daily_carbs_g`

Пересчёта на клиенте нет. Формулы остаются в онбординге и редактировании профиля.

### Fallback-поведение

Если значения норм отсутствуют (`null`), интерфейс использует безопасные fallback-значения только для отображения:
- вода: `2000`
- калории: `2000`
- белки: `120`
- жиры: `70`
- углеводы: `220`

Эти fallback-значения не записываются обратно в базу и не считаются новой нормой пользователя.

## Секция 3 — Агрегация и realtime

### Фактические данные за день

- вода: сумма `amount_ml` из `water_log` за текущий день
- еда: сумма `calories`, `protein_g`, `fat_g`, `carbs_g` из `nutrition_log` за текущий день

День определяется в логике приложения как локальный день пользователя, по тому же принципу, что и текущие today-запросы.

### Realtime-поведение

Главная подписывается сразу на:
- `water_log`
- `nutrition_log`

При изменениях в этих таблицах UI должен автоматически обновляться без перезагрузки экрана. Минимально необходимое поведение:
- подписка отслеживает `INSERT`
- после события инвалидируются соответствующие query
- Home получает актуальные агрегированные данные через обычные хуки

Это должно работать и для записей, добавленных не только из приложения, но и вручную через Supabase Dashboard.

### Хуки

#### `useWaterProgress()`

Возвращает:
- дневную сумму воды
- норму воды
- процент прогресса
- `isLoading`
- `error`

Хук отвечает за чтение `water_log`, агрегирование суммы за сегодня и подписку на realtime с последующей invalidation query.

#### realtime-слой для питания

Текущий источник фактических КБЖУ остаётся прежним — `useTodayNutrition()`. Для `Блока 1.3` вокруг него добавляется подписка на realtime, которая приводит к invalidation данных `nutrition_log` за сегодня.

Отдельная логика расчётов на главной не создаётся.

## Секция 4 — UI на главной

### Общая структура

Текущие секции воды и КБЖУ на главной удаляются и заменяются единым блоком дневного прогресса.

Состав блока:
- главное кольцо воды как основной акцент
- группа колец по питанию:
  - калории
  - белки
  - жиры
  - углеводы
- действия для добавления воды и еды

### Правила интерфейса

- вода визуально приоритетнее питания
- КБЖУ воспринимаются как единый набор дневных норм, а не как набор разрозненных карточек
- все элементы, ведущие в ещё неготовую функциональность, содержат явный текст `В разработке` или соответствующий бейдж

### Состояния

#### Loading

Показываются skeleton или приглушённые плейсхолдеры без скачков layout.

#### Empty

Если записей за день нет, кольца показывают `0 / норма`, но сам блок рендерится полностью.

#### Partial

Если часть норм отсутствует, UI остаётся стабильным: доступные кольца показывают реальные значения, для остальных используются fallback-значения отображения.

#### Error

Home не ломается. Внутри блока показывается компактное сообщение об ошибке, а структура интерфейса сохраняется.

## Секция 5 — Placeholder-экраны

### Назначение

На этом этапе создаются отдельные маршруты:
- `app/water/add.tsx` — экран добавления воды
- `app/nutrition/add.tsx` — экран добавления еды

Они нужны как честные пользовательские точки входа и как архитектурная подготовка к Блоку 2.

### Поведение

Оба экрана содержат:
- заголовок
- текст `В разработке`
- короткое пояснение, что функция скоро появится
- кнопку `Назад`

Если экран открыт напрямую, возврат должен работать корректно: назад при наличии истории, иначе переход на главную.

### Навигация

- кнопка `+` на кольце воды ведёт на placeholder-экран воды
- действие добавления еды в блоке КБЖУ ведёт на placeholder-экран еды
